## Overview
The paper tackles a timely and highly relevant systems challenge: the inefficiency of standard prefill-decode (PD) disaggregated serving when deployed for multi-round Large Language Model (LLM) workflows. Traditional PD disaggregation isolates compute-bound prefill phases and memory-bound decode phases to eliminate interference. However, multi-round workflows—such as autonomous tool-using agents and iterative Retrieval-Augmented Generation (RAG)—introduce an interleaved prefill-decode pattern. In these scenarios, the LLM generates a partial response, waits for an external tool or retrieval observation, and then requires an "incremental prefill" of the new observation before resuming decoding. 

To address this, the authors propose AMPD, an end-to-end serving framework consisting of two primary components: an offline deployment planner based on Integer Linear Programming (ILP) and an online adaptive routing mechanism. The core idea is to dynamically route incremental prefill tasks. Based on real-time Time-to-First-Token (TTFT) and Inter-Token Latency (ITL) slack metrics, AMPD either executes the incremental prefill locally on the decode worker (momentarily pausing the ongoing decode batch) or routes it remotely to a prefill worker (incurring the network overhead of transmitting the history KV cache).

## Claimed Contributions
- The identification and characterization of interleaved prefill-decode patterns in multi-round LLM workflows as a distinct challenge for existing PD disaggregation paradigms.
- The formulation of an offline deployment planner using ILP, designed to allocate resources and parallelism strategies explicitly tailored to interleaved multi-round workloads.
- The design of an online adaptive routing scheduler (AMPD) that uses a piecewise performance model and windowed TTFT/ITL statistics to dynamically balance local compute contention against remote KV cache transmission overhead.
- Empirical claims of massive improvements in Service Level Objective (SLO) attainment, ranging from 67.29% to 339.74%, over state-of-the-art co-located and disaggregated baselines.

## Novelty And Prior Work
The paper targets a critical and underexplored bottleneck in the modern ML systems literature. While recent works like DistServe, Splitwise, and Mooncake have extensively explored PD disaggregation, they largely assume a single-round request model: one large initial prefill followed by an uninterrupted stream of decoding steps. The extension of this paradigm to agentic multi-round workloads is an excellent direction. 

The novelty primarily lies in treating the placement of the *incremental prefill* as a dynamic, slack-dependent scheduling decision. Existing systems either strictly enforce the PD boundary (forcing costly network transfers of large history KV caches back and forth for every tool call) or fall back to co-located serving (re-introducing prefill-decode interference). By attempting a hybrid approach, the authors propose a pragmatic conceptual bridge. The explicit distinction between "initial" and "incremental" prefills provides a strong foundational vocabulary for future research in this domain.

## Technical Soundness
While the motivation is highly compelling, the technical execution of the adaptive routing mechanism and the offline planner presents several severe, potentially fatal flaws that undermine the theoretical soundness of the proposed system.

**1. Disregard for KV Cache Locality (Fatal Flaw Candidate):**
According to the provided pseudo-code for the adaptive routing mechanism (Algorithm 1), if the system decides against local execution, it iterates through remote prefill workers in "random order" to find a candidate with sufficient TTFT slack. For multi-round workflows, history contexts can easily span tens of thousands of tokens. By selecting remote prefill workers randomly rather than prioritizing workers that already hold a prefix-cached version of the history, the system guarantees constant, massive KV cache transfers across the network. The text acknowledges reading the history KV cache over the network (Section 3), but ignoring cache locality in the routing logic will quickly render the network a crippling bottleneck, especially at scale.

**2. Pausing Decode Batches and Tail Latency (Fatal Flaw Candidate):**
When the system routes an incremental prefill locally to avoid network overhead, it explicitly "pauses the worker's ongoing decoding batch" (Section 3). This fundamentally contradicts the primary objective of PD disaggregation, which is to isolate the memory-bound decode phase from the compute-bound prefill phase to preserve strict ITL constraints. While the system checks for ITL "slack" before making this decision, it relies on a 10-second windowed average for ITL and TTFT statistics. Using windowed averages masks severe tail latency spikes. Pausing a decode batch will almost certainly cause catastrophic violations of P99 ITL for concurrent requests co-located on that worker. The theoretical soundness of this fallback is highly questionable without mitigation strategies like chunked prefill, which are not mentioned in the scheduling logic.

**3. Static Planning for Dynamic Workloads:**
The offline deployment planner is formulated as a static Integer Linear Programming (ILP) problem based on prompt and response length estimations. However, multi-round agentic workflows are inherently non-deterministic. An agent may take two steps to solve a problem or twenty; it may receive a 50-token tool observation or a 5,000-token web scraping observation. A static offline resource allocation model is poorly suited to the high variance and unpredictability of these execution paths. The assumption that piecewise linear models ($\alpha, \beta$) can accurately capture these dynamics across heterogeneous deployment scales is a fragile foundation.

## Decision-Relevant Concerns
- **Locality-Agnostic Routing:** Iterating over prefill workers in random order guarantees worst-case network overhead for history KV cache transfers in long-context multi-round tasks.
- **Tail Latency Degradation:** Pausing ongoing decode batches to accommodate local incremental prefills reintroduces the exact interference PD disaggregation was designed to eliminate, posing severe risks to P99 ITL.
- **Brittleness of Averages:** Relying on 10-second windowed averages for scheduling decisions effectively ignores tail latency spikes, making the "slack" metrics unreliable for strict SLO compliance.
- **Static vs. Dynamic Mismatch:** The reliance on static offline ILP planning for highly variable, non-deterministic agentic workflows introduces a structural disconnect.
- **Hyperparameter Sensitivity:** The routing logic is heavily dependent on predefined threshold multipliers ($\alpha, \beta$), which likely require extensive, workload-specific manual tuning.
