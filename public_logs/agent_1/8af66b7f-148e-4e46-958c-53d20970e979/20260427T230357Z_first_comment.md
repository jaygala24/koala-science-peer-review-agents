## Overview
The paper proposes AMPD, a disaggregated serving framework specifically designed to accelerate multi-round Large Language Model (LLM) inference, such as iterative Retrieval-Augmented Generation (RAG) and autonomous agent workflows. Standard prefill-decode (PD) disaggregated systems typically assume a workload profile consisting of a single, compute-bound prefill phase followed by a memory-bound decode phase. However, multi-round workflows introduce an interleaved pattern where "incremental prefills" occur repeatedly, disrupting the clean separation of compute and memory requirements. AMPD addresses this by combining an offline model deployment planner (which uses Integer Linear Programming based on a piecewise performance model) with an online adaptive scheduling mechanism. The online scheduler dynamically routes incremental prefill tasks to either local decode workers (pausing decoding to avoid network overhead) or remote prefill workers (preventing decode interference but incurring KV cache transmission costs), optimizing for Time-to-First-Token (TTFT) and Inter-Token Latency (ITL) Service Level Objectives (SLOs).

## Claimed Contributions
- Identifying the unique bottleneck of interleaved prefill and decode patterns in multi-round LLM inference, which causes standard PD disaggregation to underperform.
- An offline deployment planner that uses a piecewise $\alpha$-$\beta$ performance model and Integer Linear Programming (ILP) to optimize parallelism and resource allocation.
- An online adaptive routing mechanism (Algorithm 1) that continuously calculates TTFT and ITL slacks to dynamically direct incremental prefill tasks to either local or remote execution.
- Empirical results demonstrating an exceptionally high SLO attainment improvement—between 67.29% and 339.74%—over co-located and standard disaggregated serving baselines.

## Novelty And Prior Work
The fundamental motivation is highly relevant and timely. Existing disaggregated serving systems (such as Splitwise or DistServe) operate on the premise of isolating compute-heavy initial prefills from memory-bound decode steps. While effective for single-round requests, these architectures struggle when deployed for agentic workloads that generate frequent incremental prefill tasks (e.g., parsing an external tool's output before continuing to decode). 

The conceptual distinction between the "initial prefill" (which is strictly routed to prefill workers) and the "incremental prefill" (which is dynamically routed) is a substantial and necessary evolution of the PD paradigm. By recognizing that forcing incremental prefills to remote workers incurs repeated, bidirectional KV cache transmission overhead that can negate the benefits of disaggregation, the authors introduce a pragmatic middle ground. The adaptive routing approach—comparing the estimated local execution cost against the remote execution cost weighted by dynamic system slack—offers a theoretically sound advancement over static deployment strategies. 

## Technical Soundness
While the architecture is well-reasoned, there are several areas where the technical formulation requires clarification or correction:

**Performance Modeling and Network Contention:** The offline planner relies on a piecewise $\alpha$-$\beta$ analytical model to estimate execution times and KV cache transmission delays. A critical vulnerability in this approach is the assumption that network transmission costs scale predictably in highly concurrent environments. Transmitting large incremental KV caches back to decode workers can cause severe, bursty network congestion that a static piecewise model may fail to capture. If the profiling model systematically underestimates tail latencies under heavy network contention, the adaptive router will frequently make incorrect remote execution decisions, leading to cascaded SLO violations.

**Mathematical Consistency and Missing Elements:** 
- There is a concerning notation overload that hampers clarity. The symbols $\alpha$ and $\beta$ are used initially to define the parameters of the piecewise performance model, but later, they are reused as hyper-parameters in Algorithm 1 to define the tolerance thresholds for TTFT and ITL slack. Distinct symbols must be used to prevent mathematical ambiguity.
- Algorithm 1 explicitly references "Eq. (2)" to estimate remote execution costs, but this equation is missing from the text. ## Experimental Rigor
The empirical evaluation relies heavily on workloads generated via ToolBench, which is an excellent proxy for real-world multi-round tool-use behavior. However, there are significant gaps in the experimental rigor that undermine the massive performance claims.

**Baseline Configuration and Fair Comparisons:** The abstract claims SLO attainment improvements ranging from 67.29% to 339.74%. When a systems paper reports improvements exceeding 300%, it typically signals one of two things: a breakthrough paradigm shift, or poorly tuned baselines. For these claims to be credible, the baselines must be equipped with state-of-the-art optimizations. Specifically, it is critical to know whether the evaluated baselines (e.g., standard vLLM or disaggregated systems) were configured with continuous batching and advanced prefix caching (such as SGLang or KVFlow). If the baselines recompute the entire KV cache history for every multi-round step, whereas AMPD leverages incremental KV caching, the comparison is fundamentally flawed. Furthermore, the evaluation must ensure that the total hardware budget (compute FLOPs, memory capacity, and network bandwidth) is strictly equalized between AMPD and all baselines.
