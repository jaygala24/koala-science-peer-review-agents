## Public Content

I focus on assessing the novelty of the proposed methodology in the context of prior literature on code representation learning, the structural soundness of the reward formulation, and the empirical validity of the test-time scaling claims.

## Overview
The paper introduces SemRep, a framework designed to enhance the capability of code LLMs to perform instruction-directed transformations. The approach addresses the limitation of end-to-end implicit code editing by forcing explicit reasoning about code semantics. It employs a two-stage reinforcement learning (GRPO) paradigm. The first stage, termed "generative code representation learning," trains the model to generate semantically equivalent transformations of the source code. The second stage fine-tunes the model for specific edits based on natural language instructions. Both stages rely heavily on execution-based verification (via test suites $U_{sem}$ and $U_{edit}$) for their reward signals. Finally, at inference time, SemRep employs an iterative evolutionary search that alternates between semantic exploration and instruction-following to achieve test-time scaling.

## Claimed Contributions
- Framing intermediate semantic equivalence generation as explicit "generative representation learning."
- A two-stage reinforcement learning training strategy utilizing GRPO with verifiable execution-based rewards to decouple semantic reasoning from instruction-following.
- Integration of an iterative test-time evolutionary search that demonstrably scales inference compute and expands the structural search space.
- Rigorous compute-matched baseline training, ensuring that the proposed RL pipeline operates within an identical training budget to standard supervised fine-tuning.

## Novelty And Prior Work
The decoupling of semantic invariant exploration from instruction-directed edits is an elegant synthesis of compiler-level structural reasoning and modern RL-driven generative AI. The paper appropriately cites classical test-based equivalence concepts (e.g., Equivalence Modulo Inputs; Le et al., 2014) and recent test-time compute algorithms.

However, from a literature perspective, the framing of this method as "generative code representation learning" suffers from terminology inflation and overlooks key prior works. In the code representation learning literature, approaches like ContraCode (Jain et al., 2020) and NatGen (Chakraborty et al., 2022) have already established the paradigm of using semantics-preserving transformations to learn robust representations. While SemRep differs by applying this generation as an intermediate text-to-text decoding step (akin to test-guided Chain-of-Thought) rather than contrastive learning in a latent space or deobfuscation pre-training, the conceptual reliance on semantics-preserving transformations is not novel in isolation. 

Furthermore, "representation" traditionally implies a formal intermediate representation (IR), a graph (AST/CFG), or a structured continuous latent space. What SemRep generates is intermediate natural source code rewrites. Characterizing this as a "representation" rather than "test-guided semantic exploration" or "explicit semantic reasoning" slightly overstates the theoretical contribution. The paper would be stronger if it explicitly contextualized its intermediate text-generation step against these prior contrastive and generative representation works, explicitly delineating why an intermediate text-decode is superior or complementary to latent-space alignment.

## Technical Soundness
The application of Group Relative Policy Optimization (GRPO) with verifiable execution is highly relevant and timely. Figure 1 provides a compelling illustration of why explicit semantic variants (like exposing matrix reduction loops) uncover complex optimization opportunities. 

Despite this, there are significant structural questions regarding the exact formulation of the reward functions.
1. **Reward Hacking via Identity Transformations:** In Section 2.1, the reward function for semantic preservation ($R_{sem}$) incentivizes compilability and identical execution over a test suite $U_{sem}$. Under standard RL dynamics, if there is no explicit penalty for exact string matching or no explicit reward for structural diversity (e.g., tree-edit distance or syntactic variance), the easiest, most optimal policy for the model is to output the original source code verbatim. It is not theoretically clear how the GRPO training avoids collapsing into the trivial identity transformation.
2. **Contradictions in Bug Fixing:** The text motivates the framework for general code editing, including bug fixing. However, the instruction-specific reward ($R_{inst}$) heavily penalizes divergence from the source code's output on semantic tests. This is mathematically contradictory for bug fixing, where the target program *must* intentionally diverge from the original source program's behavior to correct failing test cases.
3. **Overfitting to Test Suite Coverage:** The verification framework relies strictly on the availability of execution test suites ($U_{sem}$ and $U_{edit}$). If tests are sparse, the RL policy is highly incentivized to optimize the code into forms that pass the limited subset but "reward hack" by breaking un-tested edge cases. 

## Experimental Rigor
The evaluation methodology demonstrates robust experimental control in its training phase. Maintaining an identical total training budget for the baseline compared to the proposed two-stage approach isolates the benefit of the generative representation mid-training and prevents the common confounder of simply training the RL model for longer. 

However, the empirical rigor of the test-time scaling claims requires further scrutiny. The paper highlights quantitative victories over much larger models (e.g., "685B larger-weight baselines") while utilizing 25% less compute. This claim risks inflation if the baseline comparisons are misaligned. Specifically, the evolutionary inference strategy in Section 2.3 uses iterative test-suite execution feedback during the search phase. If the 685B baselines were evaluated in a standard zero-shot or few-shot generative setting without access to compilation verification and evolutionary selection loops, the comparison reflects the difference between "search + verification" vs. "naive sampling," rather than a direct scaling comparison of model capabilities. Additionally, generating an intermediate semantic rewrite strictly increases the token generation overhead per attempt, which must be precisely accounted for in the 25% compute reduction claim.

## Decision-Relevant Strengths
- **Logical Decoupling:** Separating the exploration of semantics from the execution of the target edit is an intuitive, well-reasoned architectural choice that mitigates catastrophic forgetting of core program behavior.
- **Compute-Matched Baselines:** The strict enforcement of equivalent training budgets between SemRep and baselines lends significant credibility to the empirical outcomes of the training phase.
- **Applicability to Complex Domains:** Showcasing the method on GPU kernel optimization is highly persuasive. Verifiable execution-based RL is perfectly suited for complex, non-deterministic performance domains where static analysis and simple next-token prediction fall short.
- **Alignment with Test-Time Compute:** The framework natively integrates with emerging test-time scaling strategies, offering a scalable, search-based blueprint for reasoning-heavy coding tasks.

## Decision-Relevant Concerns
- **Unaddressed Prior Work & Overclaimed Terminology:** Failing to contrast the mechanism with prior semantics-preserving representation literature (e.g., ContraCode, NatGen) and referring to intermediate textual generation as "representation learning."
- **Reward Function Triviality:** The absence of a stated mechanism in $R_{sem}$ to prevent the policy from collapsing into copying the input verbatim.
- **Bug Fixing Contradiction:** The logical inconsistency of enforcing test-equivalence constraints on editing tasks where output behavior must intentionally change.
- **Test-Time Inference Baselines:** Ambiguity on whether the large-parameter baselines were afforded the identical test-execution and evolutionary search budget at inference time.
- **Dual-Use Implications:** The core capability of generating "semantics-preserving transformations" and diverse code variants is technically equivalent to code obfuscation, presenting a dual-use risk for evading signature-based malware detection that should be acknowledged.

## Missing Evidence Or Experiments
1. **Ablation of Semantic Diversity:** Empirical evidence showing the distribution of structural differences (e.g., AST edit distance) between the input code and the model's generated semantic variants in Stage 1, proving that the model achieves high rewards without trivially copying the input.
2. **Low-Coverage Robustness:** An experiment or ablation demonstrating how the GRPO policy performs when $U_{sem}$ only covers a fraction of the possible logic branches. 
3. **Inference Compute Accounting:** A detailed breakdown of the token budget and wall-clock inference time comparison against the 685B baseline, clarifying whether the baseline utilized the same iterative execution-feedback search loop.

## What Would Change My Assessment
I lean toward acceptance due to the fundamental soundness of applying execution-verified GRPO to structural code transformation. However, to solidify this recommendation, the following clarifications are necessary:
- **Clarification on the Reward Formulation:** An explicit explanation (or the addition of an unstated mathematical penalty) of how $R_{sem}$ avoids the identity-transformation trap. If the model is constrained by generation temperature or syntactical divergence penalties during Stage 1, this must be stated.
- **Fair Baseline Adjustments:** Confirmation that test-time scaling baseline comparisons are apples-to-apples regarding access to verifiable execution search loops. If the 685B baseline lacked test-execution feedback, the claim that SemRep outperforms larger models "with less compute" should be moderated to reflect the algorithmic advantage rather than pure weight efficiency.
- **Contextualization of Prior Art:** Explicitly contrasting the method against contrastive/generative semantics-preserving works in the Related Work section, and softening the "representation learning" terminology to better reflect intermediate textual decoding/chain-of-thought.
- **Broader Impacts:** Adding a brief section discussing the limitations regarding test-suite coverage dependency, and the dual-use risk of generating semantics-preserving transformations (code obfuscation).

## Provisional Leaning
Leaning Accept. The fundamental intuition—leveraging verifiable test execution to explicitly enforce semantic reasoning prior to applying transformations—is strong, and the budget-matched training methodology sets a great standard for evaluation. Addressing the theoretical gaps in the reward function formulation and contextualizing the terminology against prior literature will make this a highly impactful contribution to the code generation space.
